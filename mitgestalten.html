Beitragen

Eigene Templates

GitHub

Diese Anleitung richtet sich an Nutzende, die eigene Quarto-Templates auf GitHub bereitstellen möchten. In der Begrifflichkeit von Quarto nutzen wir hier Starter-Templates.

Voraussetzungen

Auf Ihrem Rechner:

  • GitHub-Account
  • Git installiert
  • Quarto installiert
  • ein lokales Verzeichnis mit Ihrem Starter-Template (vgl. Schritt 1 unten)

Auf GitHub:

  • Berechtigung, Repositories anzulegen (im persönlichen Account oder in einer Organisation)
  • bei Organisationsnutzung: passende Berechtigungen im Ziel-Team bzw. in der Organisation
Template vorbereiten
  • Lokales Verzeichnis: Ihr Ordner auf dem Rechner, der die Template-Dateien enthält.
  • Repository auf GitHub: das Remote-Repository, in dem Sie dieses Template veröffentlichen.

Schritt 1: Starter-Template im lokalen Verzeichnis vorbereiten

Legen Sie in Ihrem lokalen Verzeichnis genau die Dateien ab, die beim Erzeugen neuer Projekte aus dem Template kopiert werden sollen.

Wichtig: Quarto kopiert bei Starter-Templates grundsätzlich fast alles aus dem Repository, jedoch typischerweise nicht:

  • versteckte Dateien/Ordner, die mit . beginnen (z. B. .gitignore)
  • typische Repository-Dateien wie README.md und LICENSE

Das ist hilfreich, weil Sie so Dokumentation und Lizenz im Template-Repository pflegen können, ohne dass sie bei jeder Template-Nutzung automatisch im erzeugten Projekt landen.

Empfehlung:

  • Halten Sie die Template-Dateien bewusst minimal.
  • Nutzen Sie neutrale Platzhalter statt fachlicher Beispielmaterialien.

Schritt 2: Lokales Verzeichnis als Git-Repository initialisieren

Wechseln Sie in das lokale Verzeichnis und initialisieren Sie Git:

git init

Erstellen Sie einen ersten Commit:

git add .
git commit -m "Initial commit (starter template)"

Schritt 3: Repository auf GitHub anlegen

Ab hier unterscheidet sich das Vorgehen je nach Ziel:

Variante A: Persönliches Repository

Variante B: Repository in einer Organisation (mit Teams)

Die technischen Schritte sind ähnlich, aber Owner, Sichtbarkeit und Berechtigungen unterscheiden sich.

Variante A: Persönliches Repository

Schritt A1: Leeres Repository im persönlichen Account erstellen

Erstellen Sie auf GitHub ein neues Repository unter Ihrem persönlichen Account.

Empfehlungen:

  • vergeben Sie einen sprechenden Namen (z. B. quarto-template-)
  • setzen Sie eine klare Kurzbeschreibung (was das Template erzeugt)
  • initialisieren Sie das Repository nicht mit README/License, weil diese lokal bereits vorhanden sein können

Schritt A2: Lokales Repository mit dem GitHub-Repository verknüpfen

Fügen Sie die Remote-URL hinzu:

git remote add origin https://github.com/<username>/<repo>.git

Setzen Sie den Standard-Branch (empfohlen: main) und pushen Sie:

git branch -M main
git push -u origin main

Schritt A3: Repository als Template aktivieren

Aktivieren Sie im GitHub-Repository die Template-Funktion, damit Nutzende über „Use this template“ ein neues Repository daraus erstellen können.

Optional (aber hilfreich): Ergänzen Sie Topics wie quarto und template, damit das Repository auffindbarer ist.

Schritt A4: Nutzung testen

Testen Sie lokal, ob Quarto das Starter-Template korrekt zieht:

quarto use template <username>/<repo>

Damit sehen Sie unmittelbar, welche Dateien in ein neues Projekt kopiert werden.

Variante B: Repository in einer Organisation (mit Teams)

Schritt B1: Leeres Repository in der Organisation erstellen

Erstellen Sie das Repository unter der Organisation als Owner.

!!!! Ausführlicher !!!

Achten Sie dabei insbesondere auf:

  • Owner: Organisation (nicht Ihr persönlicher Account)
  • Sichtbarkeit: public/internal/private entsprechend Ihrer Zielgruppe
  • Zugriff: Team(s) mit Write/Admin-Rechten, die das Template pflegen sollen

Hinweis:

  • Wenn das Template breit nachgenutzt werden soll, ist ein öffentliches Repository meist am unkompliziertesten. -In Organisationen wird der Zugriff üblicherweise über Teams gesteuert.

Schritt B2: Lokales Repository mit dem Organisations-Repository verknüpfen

Setzen Sie die Remote-URL auf die Organisationsadresse:

git remote add origin https://github.com/<org>/<repo>.git

Pushen Sie wie oben:

git branch -M main
git push -u origin main

Schritt B3: Template-Funktion aktivieren und Team-Zugriffe prüfen

Aktivieren Sie im Repository die Template-Funktion, damit Teams und externe Nutzende (je nach Sichtbarkeit) das Repository als Vorlage verwenden können.

Prüfen Sie anschließend:

  • Welche Teams dürfen Issues bearbeiten?
  • Wer darf Pull Requests mergen?
  • Wer darf Releases/Tags erstellen (falls Sie Versionierung nutzen)?

Empfehlung:

  • Legen Sie klar fest, welches Team für Reviews und Releases verantwortlich ist.
  • Dokumentieren Sie den Beitragspfad im Repository (z. B. im README oder in CONTRIBUTING.md).

Schritt B4: Nutzung testen

Testen Sie auch hier lokal:

quarto use template <org>/<repo>

Wenn das Repository nicht öffentlich ist, benötigen Nutzende passende GitHub-Rechte, damit der Zugriff funktioniert.

Schritt 4: Minimale Nutzungsanleitung im Template-Repository ergänzen

Auch wenn README.md typischerweise nicht in neue Projekte kopiert wird, ist sie für Ihre Nutzenden im Template-Repository zentral.

Empfehlung für Inhalte der README:

  • Zweck des Starter-Templates
  • wie das Template genutzt wird (quarto use template … und/oder „Use this template“)
  • welche Dateien beim Erzeugen neuer Projekte kopiert werden (und welche nicht)
  • wie Beiträge eingebracht werden (Issues, Pull Requests)
